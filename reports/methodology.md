# P&L Explain / Exception Dashboard  
## Methodology and Controls Document

## 1. Objective

This project simulates a middle-office workflow for a small spot FX trading book. The objective is to calculate expected daily P&L, compare it to reported P&L from a separate simulated source, identify material variances, classify likely root causes, and present the results in a dashboard-style reporting layer.

The project is designed to demonstrate operationally relevant skills in:
- daily P&L production
- variance analysis
- exception detection
- investigation support
- controls documentation
- Python-based reporting workflows

---

## 2. Scope

This project covers a single simulated spot FX trading book across three USD-quoted currency pairs:

- EURUSD
- GBPUSD
- AUDUSD

The sample covers 45 business days of activity and includes:
- daily market prices
- executed trades
- transaction fees
- derived positions
- expected P&L
- reported P&L with deliberate breaks
- rule-based exception classification

The reporting frequency is daily.

---

## 3. Project Workflow

The project follows the workflow below:

1. Generate clean source-of-truth data
2. Calculate expected daily P&L
3. Create a simulated reported P&L source with planted breaks
4. Compare expected and reported P&L
5. Flag material variances
6. Classify likely causes
7. Produce dashboard tables and charts
8. Document selected investigations

## 5. Source-of-Truth Design

The clean operational source-of-truth files are:

- `positions.csv`
- `trades.csv`
- `fees.csv`
- `prices.csv`

Expected P&L is derived from these files.

`reported_pnl.csv` is treated as a separate downstream reporting source used for variance analysis. It is intentionally modified to contain simulated reporting breaks for exception testing.

`break_scenario_log.csv` is used only for validation of classification performance and is not used by the exception engine as an input.

---

## 6. Expected P&L Methodology

### 6.1 Overview

Expected daily P&L is calculated by instrument and date using inventory state, weighted average cost, end-of-day prices, and trade-level fees.

The project tracks:
- running open position
- weighted average cost
- realized P&L from position reductions or closures
- change in unrealized P&L
- transaction fees

### 6.2 Formula

Expected total daily P&L is defined as:

**Expected Total P&L = Realized P&L + Change in Unrealized P&L - Fees**

Where:

- **Realized P&L** is generated when a trade closes or reduces an existing position
- **Beginning Unrealized P&L** reflects the start-of-day mark on the existing position
- **Ending Unrealized P&L** reflects the end-of-day mark on the updated position
- **Change in Unrealized P&L = Ending Unrealized P&L - Beginning Unrealized P&L**
- **Fees** are the sum of transaction fees for the date/instrument

### 6.3 Position sign convention

- BUY trades increase position
- SELL trades decrease position
- Long positions are positive
- Short positions are negative

### 6.4 Realized P&L treatment

Realized P&L is generated only when a trade reduces an existing position.

Examples:
- selling against an existing long position
- buying back an existing short position

If a trade increases exposure in the same direction, no realized P&L is booked and weighted average cost is recalculated.

### 6.5 Attribution fields

In addition to total expected P&L, the project stores:
- `realized_pnl`
- `unrealized_change_pnl`
- `fees`
- `position_pnl`
- `trade_day_pnl`

These fields support dashboard reporting and investigation analysis.

## 7. Reported P&L Construction

Reported P&L is generated by starting from expected P&L and then applying controlled simulated breaks. This creates a realistic second source against which expected P&L can be compared.

The following exception types are injected:

- missing trade
- incorrect fee
- stale price
- duplicate booking
- wrong position

A separate `break_scenario_log.csv` file records which breaks were inserted, their magnitude, and supporting metadata.

---

## 8. Exception Detection Methodology

### 8.1 Variance calculation

For each date and instrument:

**P&L Variance = Reported P&L - Expected P&L**

The project also calculates:
- `abs_pnl_variance`
- `is_exception`
- `severity`

### 8.2 Materiality threshold

A row is flagged as an exception if:

**Absolute P&L Variance > $150**

### 8.3 Severity buckets

- **Low:** > $150 and <= $300
- **Medium:** > $300 and <= $750
- **High:** > $750

---

## 9. Exception Classification Rules

The exception engine applies rule-based logic to infer likely root cause.

### 9.1 Stale price
The engine tests whether the variance approximately matches:

**End Position × (Prior Close - Current Close)**

If yes, the row is classified as `stale_price`.

### 9.2 Duplicate booking
The engine tests whether the variance approximately matches the same-day P&L contribution of an individual trade.

If yes, the row is classified as `duplicate_booking`.

### 9.3 Missing trade
The engine tests whether the variance approximately matches the negative of one trade’s same-day P&L contribution.

If yes, the row is classified as `missing_trade`.

### 9.4 Wrong position
The engine tests whether the variance can be explained by a plausible round-lot quantity mismatch multiplied by the daily price move.

If yes, the row is classified as `wrong_position`.

### 9.5 Incorrect fee
If the row contains trades, the variance is moderate, and the variance is not well explained by stronger price or trade-booking rules, the row is classified as `incorrect_fee`.

### 9.6 Unknown
If the variance is material but not well explained by the current rule set, the row is classified as `unknown`.

---

## 10. Validation Approach

The exception engine operates independently from the planted break log.

After classification is complete, the results are compared against `break_scenario_log.csv` to evaluate classification accuracy.

In this project build, the rule-based exception engine correctly identified 100% of the planted exceptions and matched the planted root cause labels.

## 11. Control Checks

The project includes the following data and process controls.

### 11.1 Input data controls
- trade IDs must be unique in the clean trade file
- every trade must have exactly one fee record
- prices must have one row per date and instrument
- positions must have one row per date and instrument
- dates must follow business-day sequence
- instruments must use consistent naming

### 11.2 Position continuity control
For each instrument:

**Next day start position = prior day end position**

This verifies inventory continuity through time.

### 11.3 Position roll-forward control
For each date and instrument:

**End Position = Start Position + Net Signed Trade Quantity**

This verifies that the position file is consistent with the trade file.

### 11.4 Fee completeness control
Every trade must map to one fee record.

### 11.5 Price completeness control
Each date/instrument row must contain both:
- `close_price`
- `prior_close_price`

### 11.6 Expected vs reported variance control
Rows where absolute variance exceeds materiality threshold are flagged for investigation.

### 11.7 Classification support fields
The exception report stores supporting fields such as:
- stale price candidate delta
- matched trade ID
- matched trade impact
- implied position delta
- investigation note

These fields improve reviewability and auditability.

## 13. Assumptions

The project makes the following assumptions:

- the book contains only spot FX instruments
- all P&L is represented in USD terms
- one book is supported
- one end-of-day close is used for daily marking
- fee logic is simplified relative to real institutional schedules
- exception classification uses rule-based logic rather than machine learning
- the reported P&L source is simulated rather than produced by a separate live system

---

## 14. Limitations

This project is intentionally simplified for clarity and scope control.

Key limitations include:
- no intraday timestamps
- no settlement workflow
- no counterparty dimension
- no trader IDs or desk hierarchy
- no multi-book allocation logic
- no partial-fill logic
- no FX translation layer beyond the simplified USD setup
- no derivatives greeks or complex product-control logic
- no live market or broker connectivity

These simplifications were chosen to keep the project focused on daily P&L explain and exception handling rather than full production trade-support infrastructure.
